# Capacitor 커스텀 플러그인 기초 및 MultiWebView 개발 원리

이 문서는 Capacitor 환경에서 네이티브 기기 기능을 제어하는 커스텀 플러그인을 만들고, 서브 웹뷰(Multi WebView) 통신을 구현하여 Vue 프로젝트에서 활용하는 전체 과정과 핵심 원리를 설명합니다. 이 내용을 이해하면 새로운 네이티브 기능을 추가하거나 기존 플러그인을 수정(Customizing)할 수 있습니다.

---

## 1. Capacitor 플러그인의 기본 원리

Capacitor 앱 구조는 크게 두 부분으로 나뉩니다.
1. **웹 영역 (JavaScript/Vue)**: 우리가 만드는 UI 화면 (`index.html`, Vue 컴포넌트)
2. **네이티브 영역 (Java/Kotlin/Swift)**: 안드로이드나 iOS의 부품들을 제어하는 코어 계층

웹 브라우저(JavaScript)는 기본적으로 네이티브 기기의 자원(카메라, 진동, 새로운 네이티브 창 띄우기 등)에 직접 접근할 수 없습니다. 이를 중간에서 번역하고 다리를 놔주는 역할이 바로 **Bridge(브릿지)** 이며, 커스텀 플러그인을 만든다는 것은 이 다리의 **웹 측 스위치(Interface)**와 **네이티브 측 실행기(Implementation)**를 한 쌍으로 만드는 작업입니다.

### 플러그인 동작 흐름
`Vue 컴포넌트` -> `Plugin Interface (TypeScript)` -> *[Capacitor Bridge]* -> `Native Plugin 로직 (Kotlin/Swift)`

## 2. MultiWebView 플러그인의 핵심 원리

서브 웹뷰 추가 플러그인은 "안드로이드 네이티브 화면 위에 새로운 미니 브라우저(WebView)를 하나 더 얹는 것"입니다.

### 핵심 동작 원리 4가지

1. **웹뷰 인스턴스 관리 (Map)**
   - 한 개 이상의 서브 웹뷰를 띄울 수 있으므로, 웹뷰 객체를 ID(문자열)를 키(Key)로 하는 맵(Map / Dictionary)에 저장하고 관리해야 합니다. 
   - 그래야 특정 ID를 지정해서 웹뷰를 닫거나, 특정 웹뷰에만 메시지를 보낼 수 있습니다.

2. **UI 스레드 (Main Thread) 동기화**
   - 안드로이드/iOS 모두 **"화면에 무언가를 그리는 작업"은 반드시 메인(UI) 스레드에서만 실행**되어야 한다는 엄격한 규칙이 있습니다.
   - 자바스크립트에서 보낸 명령은 백그라운드 스레드에서 도착하므로, 코드를 반드시 `activity.runOnUiThread { ... }` 로 감싸서 메인 스레드에 화면 그리기 작업을 예약해야 앱이 죽지 않습니다.

3. **양방향 통신 브릿지 (가장 중요)**
   - **웹 -> 네이티브 (일반 PluginMethod)**: Vue 쪽에서 `MultiWebView.openWebView(...)` 처럼 호출하면 네이티브의 `@PluginMethod fun openWebView(...)`가 실행됩니다.
   - **자식 웹뷰 -> 네이티브 -> 부모 웹뷰 (이벤트 릴레이)**: 
     - 자식 웹뷰 안에는 Capacitor 코어가 없으므로 일반적인 PluginMethod를 쓸 수 없습니다.
     - 대신, 안드로이드의 `addJavascriptInterface` 기능을 이용해 자식 웹뷰의 자바스크립트 `window` 객체에 안드로이드 클래스 인스턴스(`NativeBridge`)를 강제로 주입합니다.
     - 자식 웹뷰에서 `window.NativeBridge.postMessage()`를 호출하면, 네이티브 코드가 이를 받아 Capacitor의 `notifyListeners` 스피커를 통해 부모(메인) 웹뷰로 방송합니다.
   - **부모 -> 네이티브 -> 자식 웹뷰 (콜백 호출)**: 
     - 부모가 네이티브에 명령을 내리면, 네이티브는 해당 자식 웹뷰 객체를 찾아서 `evaluateJavascript("window.onNativeMessage(...)")`를 실행합니다. 즉, 네이티브가 강제로 자식 웹뷰의 자바스크립트 함수를 실행시키는 방식입니다.

4. **자원 해제 방지 전략 (Memory Leak)**
   - 웹뷰를 닫을 때는 그냥 화면에서 숨기는 것(`setVisibility`)이 아니라, 부모 레이아웃에서 뷰를 완전히 뜯어내고(`removeView`), 내부 엔진을 파괴해 줘야(`destroy()`) 메모리 누수와 앱 크래시를 막을 수 있습니다.

## 2-1. iOS의 경우 어떤 기반을 사용하는가?

iOS에서 이와 동일한 플러그인을 만든다면, 안드로이드의 `WebView` 대신 Apple의 강력하고 최신 엔진인 **`WKWebView`**를 기반으로 구축하게 됩니다.

iOS 플랫폼에서의 핵심 원리 매핑은 다음과 같습니다:

1. **기본 뷰 부착 방식 (View-based)**
   - 안드로이드가 `activity.window.decorView`에 뷰를 더했던 것처럼, iOS는 현재 활성화된 화면인 **`CAPBridgeViewController`의 루트 `view`에 `WKWebView` 인스턴스를 서브뷰(Subview)로 추가(`addSubview`)**하는 형태가 됩니다.
   - iOS 역시 가장 가볍고 직관적인 이 *View 직접 부착 방식*을 주로 사용합니다.

2. **UI 스레드 동기화**
   - 안드로이드의 `runOnUiThread` 대신, iOS는 메인 디스패치 큐인 **`DispatchQueue.main.async { ... }`**를 사용하여 `WKWebView`를 생성하고 UI에 붙입니다.

3. **양방향 통신 브릿지 (iOS Style)**
   - **자식 웹뷰 -> 네이티브**: 안드로이드의 `addJavascriptInterface` 대신, iOS는 **`WKScriptMessageHandler`** 프로토콜 기반의 **`WKUserContentController`**를 사용합니다.
     - 네이티브(Swift) 측에서 `userContentController.add(self, name: "NativeBridge")` 와 같이 등록해 두면,
     - 웹뷰(JS) 측에서는 `window.webkit.messageHandlers.NativeBridge.postMessage(...)` 형태로 이벤트를 네이티브로 보낼 수 있습니다.
   - **네이티브 -> 자식 웹뷰**: 동작 원리는 같습니다. `WKWebView.evaluateJavaScript(...)` 메서드를 사용해 현재 띄워진 웹뷰 속에 자바스크립트 구문을 강제로 실행시켜 응답을 내려 보냅니다.

4. **로컬 서버 인터셉터 (ERR_NAME_NOT_RESOLVED 방지)**
   - 안드로이드에서 Capacitor의 `localServer` 인터셉터를 연결했듯, iOS의 Capacitor는 Scheme Handler(예: `capacitor://`)를 사용하여 로컬 파일들을 서빙합니다. 
   - iOS 구현 시 서브 웹뷰를 생성할 때 **Capacitor 브릿지가 사용하는 설정(Configuration)**을 그대로 전달하여 인스턴스화하면 정상적인 도메인 매핑이 이루어집니다.

## 3. 구현 단계별 코드 파보기

### 단계 3-1: 플러그인 인터페이스 선언 (`src/definitions.ts`)

플러그인과 웹(Vue)이 서로 약속하는 "설명서"입니다. 이 파일을 기반으로 데이터 타입과 함수 이름이 결정됩니다.

```typescript
import type { PluginListenerHandle } from '@capacitor/core';

export interface MultiWebViewPlugin {
  // 웹에서 네이티브 함수를 호출하는 명령
  openWebView(options: { url: string; webviewId: string }): Promise<void>;
  closeWebView(options: { webviewId: string }): Promise<void>;
  postMessageToWebView(options: { webviewId: string; data: any }): Promise<void>;

  // 네이티브에서 웹으로 이벤트를 방송할 때, 웹이 이를 구독(리스너)하는 규약
  addListener(
    eventName: 'onSubWebViewMessage',
    listenerFunc: (payload: { webviewId: string; data: any }) => void,
  ): Promise<PluginListenerHandle>;
}
```

### 단계 3-2: Android 네이티브 구현 (`android/src/.../MultiWebViewPlugin.kt`)

실제 하드웨어(화면)에 새 웹뷰를 그리는 부분입니다. 수정이나 기능을 추가하려면 이 파일의 구조를 이해해야 합니다.

```kotlin
@CapacitorPlugin(name = "MultiWebView")
class MultiWebViewPlugin : Plugin() {
    private val webViews = mutableMapOf<String, WebView>() // 1. 관리 명부 구조

    @PluginMethod // 웹에서 호출할 수 있도록 열어두는 어노테이션
    fun openWebView(call: PluginCall) {
        val url = call.getString("url") ?: return call.reject("URL 없음")
        val webviewId = call.getString("webviewId") ?: return call.reject("ID 없음")

        activity.runOnUiThread { // 2. UI 스레드 진입
            val newWebView = WebView(context)
            /* ...설정 코드 생략... */

            // ⚠️ 여기서 커스텀 WebViewClient를 넣지 않으면 "ERR_NAME_NOT_RESOLVED" 로드 에러가 납니다.
            // Capacitor의 로컬 서버(http://localhost) 인터셉터 로직을 자식 웹뷰도 공유해야 하기 때문입니다.
            newWebView.webViewClient = object : WebViewClient() {
                override fun shouldInterceptRequest(...) = bridge.localServer.shouldInterceptRequest(request)
            }

            // 3. 네이티브 객체를 자식 웹뷰의 JS 'window.NativeBridge' 이름으로 심어줌
            newWebView.addJavascriptInterface(SubWebViewBridge(webviewId), "NativeBridge")
            
            /* 루트 뷰에 추가 및 URL 로드 로직 */
            
            call.resolve() // 웹 쪽의 Promise를 성공(then) 상태로 반환
        }
    }

    // 통신 수신기 클래스 (웹뷰 인터페이스)
    inner class SubWebViewBridge(private val webviewId: String) {
        @JavascriptInterface // 자식 웹뷰의 JS에서 직접 접근 가능한 함수
        fun postMessage(messageData: String) {
            val ret = JSObject().apply { put("webviewId", webviewId); put("data", messageData) }
            // 메인 웹뷰 쪽으로 이벤트 전파 (addListener로 수신 가능)
            notifyListeners("onSubWebViewMessage", ret) 
        }
    }
}
```

## 4. Vue 프로젝트에서 플러그인 통합 및 응용 방법

플러그인이 완성되었다면, 루트 프로젝트(`package.json`의 `dependencies`에 플러그인 로컬 폴더 경로를 추가하고 `npm install` 후 `npx cap sync`)에서 다음과 같이 활용합니다.

### 메인 웹뷰 (부모) 역할 (`MainView.vue`)

부모 뷰에서는 `registerPlugin`으로 플러그인을 불러오고 이벤트를 구독합니다.

```vue
<script setup lang="ts">
import { registerPlugin } from '@capacitor/core'
import type { MultiWebViewPlugin } from 'capacitor-multi-webview'

// 플러그인 등록
const MultiWebView = registerPlugin<MultiWebViewPlugin>('MultiWebView')

// 1. 자식의 이벤트 수신 설정 (컴포넌트 로드 시점)
onMounted(async () => {
    await MultiWebView.addListener('onSubWebViewMessage', (payload) => {
        console.log("자식으로부터 메시지가 도착했습니다: ", payload.data)
        // 화면 상태 업데이트 로직 작성...
    })
})

// 2. 서브 웹뷰 창 열기
const openChild = async () => {
    await MultiWebView.openWebView({ 
        url: `/sub-webview-test`, 
        webviewId: 'child-1' 
    })
}

// 3. 열려있는 자식 창에게 데이터 보내기
const sendData = async () => {
    await MultiWebView.postMessageToWebView({ 
        webviewId: 'child-1', 
        data: '안녕 자식 웹뷰!' 
    })
}
</script>
```

### 서브 웹뷰 (자식) 역할 (`SubWebViewTest.vue`)

서브 웹뷰 페이지는 일반적인 **웹 환경처럼 보이지만, 네이티브가 강제로 주입한 환경 변수**들을 바탕으로 소통합니다. 따라서 일반 웹 브라우저에서는 이 로직이 작동하지 않습니다. (앱 패키징 후 앱 내에서 띄워졌을 때만 작동)

```vue
<script setup lang="ts">
import { onMounted } from 'vue'

onMounted(() => {
  // 1. 부모가 네이티브를 통해 보내는 메시지를 받을 그릇(함수)을 전역 window 객체에 열어둡니다.
  // 네이티브 플러그인은 evaluateJavascript를 통해 이 함수를 강제 실행시킵니다.
  ;(window as any).onNativeMessage = (data: any) => {
    console.log('부모 웹뷰로부터 받은 메시지:', data)
  }
})

// 2. 부모에게 메시지를 보낼 때 호출할 함수
const messageToParent = () => {
  // 네이티브가 심어놓은 'NativeBridge' 브릿지 객체가 있는지 확인한 후 호출합니다.
  if ((window as any).NativeBridge) {
    ;(window as any).NativeBridge.postMessage('안녕하세요 부모님!')
  }
}

// 3. 서브 웹뷰 자체를 닫고 싶을 때
const closeMe = () => {
  if ((window as any).NativeBridge) {
    // 부모에게 닫으라는 행동(action)을 지시하는 이벤트를 방출합니다.
    ;(window as any).NativeBridge.postMessage(JSON.stringify({ action: 'close' }))
  }
}
</script>
```

## 5. 실전: 추가 커스텀 기능 만들기

나중에 **"특정 자식 웹뷰의 크기를 화면의 절반만 차지하게 하고 싶다"** 거나 **"투명한 배경을 적용하고 싶다"**면 어떻게 해야 할까요?

1. **인터페이스 스펙 정의 (`definitions.ts`)**:
   `openWebView` 의 인자(`options`)에 `width?: number`, `height?: number` 같은 속성을 추가합니다.
2. **Android 플러그인 로직 수정 (`MultiWebViewPlugin.kt`)**:
   `openWebView` 함수 안에서 `call.getInt("width")` 값을 읽어온 뒤, 기존에 `MATCH_PARENT`로 주어졌던 `FrameLayout.LayoutParams` 영역을 수정합니다.
   투명하게 만들고 싶다면 `newWebView.setBackgroundColor(Color.TRANSPARENT)` 메소드를 찾아 레이아웃 로직에 주입합니다.
3. **웹뷰 빌드 및 싱크**: 
   플러그인 폴더 내에서 `npm run build` 스크립트를 재실행하고, 루트 프로젝트에서 `npx cap sync android`를 실행하여 플러그인 변경사항을 안드로이드 스튜디오 쪽으로 밀어 넣은 뒤 실행하면 동작합니다.
