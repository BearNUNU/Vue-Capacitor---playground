# 🛠️ 프로젝트 이슈 및 해결 방안

## 1. 프로젝트 경로에 한글 포함 시 빌드 오류 발생
- **이슈 사항:** 
  프로젝트 경로에 한글이 포함되어 있을 경우, `npx cap open` 실행 시 자바로 앱을 빌드하는 과정에서 오류가 발생합니다.
- **해결 방안:** 
  프로젝트 경로 및 폴더명에는 **영어만 사용**하도록 합니다.

---

## 2. Xcode 빌드 권한 오류 (`Contents.json`)
- **이슈 사항:** 
  Xcode 빌드 시 `The file “Contents.json” couldn’t be opened because you don’t have permission to view it.`라는 오류가 발생합니다. macOS가 기본적으로 터미널이나 VS Code 등에서 바탕화면 폴더를 임의로 읽고 쓰는 것을 차단하기 때문에 발생합니다.
- **해결 방안:** 
  맥의 좌측 상단 ` (애플 로고)` > **시스템 설정(System Settings)**을 엽니다.
  **개인정보 보호 및 보안(Privacy & Security)** 메뉴로 들어갑니다.
  **파일 및 폴더(Files and Folders)** 항목을 클릭합니다.
  사용 중인 터미널/에디터의 **데스크탑 폴더 스위치를 on**으로 변경하여 권한을 부여합니다.

---

## 3. 로컬 환경에서 HTTP 통신 자동 차단 (Android)
- **이슈 사항:** 
  Android API 28 (안드로이드 9) 이후부터는 HTTP(Cleartext) 통신을 기본적으로 막고 있어서, 로컬 기반 서버(HTTP)를 사용할 시 통신이 자동 차단되는 문제가 발생합니다.
- **해결 방안:** 
  1. **Capacitor 설정 업데이트**: `capacitor.config.ts` 파일에서 `server` 객체 내부에 `cleartext: true` 속성을 추가합니다.
  2. **Android Manifest 파일 수정**: 안드로이드 `AndroidManifest.xml`의 `<application>` 태그 내에 `android:usesCleartextTraffic="true"` 속성을 추가합니다.

---

## 4. 서브 웹뷰에서 백엔드로 요청 시 세션(HttpOnly) 쿠키 누락 문제
- **이슈 사항:** 
  커스텀 플러그인(`MultiWebViewPlugin`)을 통해 띄워진 서브 웹뷰에는 `CapacitorHttp` 통신 브릿지가 적용되지 않습니다. 브라우저의 기본 `fetch()`가 활성화되며, 로컬 환경에서 다른 외부 아이피나 포트로 요청 통신 시(`http://localhost` ➔ `http://192.168.x.x`) 최신 크롬(Android 브라우저)의 **Cross-Origin(SameSite=Lax) 보안 정책**이 발동됩니다. 이로 인해 HTTP 기반에서는 세션 쿠키 전송이 차단(Drop)되어 401 인증 에러가 발생합니다.
- **해결 방안:** 
  - **(로컬 개발망 임시 우회)**
    안드로이드 Capacitor 내부 통신 인터셉터 경로를 사용합니다. 서브 웹뷰 내에서 서버로 API 호출 시 목적지를 `http://localhost/_capacitor_http_interceptor_?u={백엔드주소}` 형태로 변경하면, 브라우저가 직접 서버를 찌르는 것이 아니라 **네이티브 안드로이드 소켓(`HttpURLConnection`) 플러그인**을 통해 통신하므로 SameSite 쿠키 제약을 우회하여 통신할 수 있습니다.
  - **(상용 서버 필수 반영 방안)**
    실제 상용 런칭 시에는 백엔드 통신 구간에 필수적으로 **HTTPS 5T SSL 통신**을 적용하고, 세션 유지 쿠키에 `SameSite=None; Secure` 속성을 설정해야만 브라우저 표준에 부합하게 보안 차단 없이 HTTP 쿠키가 전송됩니다.

## 5. 로그인 후 스플래시 페이지(alert 창)가 다이얼로그처럼 뜨는 이슈
- **이슈 사항:** 
  로그인 성공 등 웹 화면 내부에서 자바스크립트의 기본 `alert()` 함수를 호출하게 되면, Capacitor가 이를 가로채서 백그라운드가 딤 처리된 거대한 안드로이드 네이티브 팝업창(`AlertDialog`, OK 버튼 포함)으로 띄웁니다. 이 과정 중 테마 설정과 맞물려 화면 전체를 덮게 되면서, 사용자는 이를 의도되지 않은 스플래시 이미지나 버그 화면으로 오인하게 됩니다.
- **해결 방안:** 
  1. **브라우저 네이티브 브릿지 오버라이드 및 전역 모듈화**:
     Vue 페이지마다 `isApp()` 등으로 분기 처리하는 것을 방지하기 위해, 웹과 앱의 UI/UX 인터페이스(예: `window.alert()`, `window.confirm()`)를 가로채어 덮어쓰는(Overriding) **전용 전역 브릿지 관리 모듈(예: `src/utils/nativeOverrides.ts`)을 별도로 분리하여 작성**합니다. 이 모듈을 앱의 진입점(`main.ts`)에서 한 번만 Import 및 초기화 하면, 컴포넌트 내부(비즈니스 로직)에서는 단순히 `alert()`만 호출해도 PC웹에서는 브라우저 팝업이, 모바일 앱에서는 자체 네이티브 브릿지 다이얼로그가 뜨도록 완벽하게 추상화하고 유지보수를 용이하게 할 수 있습니다.
